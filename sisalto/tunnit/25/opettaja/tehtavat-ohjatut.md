# Ohjatut tehtävät — oppitunti 25

Tunnin tehtävissä ihminen sijoitetaan kohtiin, joissa päätös todella tarvitsee harkintaa ja vastuuta. Oppijoiden ensimmäinen ratkaisu on usein hyväksyttää kaikki tai jättää ihminen kokonaan prosessin ulkopuolelle. Kuljeta keskustelua näiden ääripäiden väliin kysymällä aina, kuinka vakava virhe olisi, voiko toiminnon perua ja kuka joutuu perustelemaan päätöksen jälkikäteen.

## Aktiviteetti 1 — Missä ihminen tarvitaan? (15 min)

Luokittele tilanteet:

- tavallisen vastaanottokuittauksen lähettäminen
- hinnanalennuksen hyväksyminen
- tietokannan rivin poistaminen
- lisätiedon pyytäminen viestiin, josta puuttuu pakollinen tieto
- lääkinnälliseen tai oikeudelliseen päätökseen vaikuttava suositus

Kysy jokaisesta: mikä on virheen vaikutus, voidaanko toiminto perua ja kuka kantaa vastuun?

Älä hae yhtä kaikille tilanteille sopivaa luokitusta. Vastaanottokuittaus voi yleensä lähteä automaattisesti, kun taas peruuttamaton poisto tai merkittävä rahallinen sitoumus tarvitsee portin. Tärkeintä on, että ryhmä yhdistää ihmisen osallistumisen tunnistettavaan riskiin eikä epämääräiseen varovaisuuteen.

## Aktiviteetti 2 — Rakenna päätösnäkymä (15 min)

Hyväksyntäportissa pitää näkyä:

- mitä päätöstä pyydetään
- mihin aineistoon ehdotus perustuu
- agentin lyhyt havaittava perustelu
- riskitaso ja seuraukset
- hyväksy, muokkaa, hylkää ja eskaloi -vaihtoehdot

Anna ryhmille tarkoituksella puutteellinen hyväksyntäviesti ja pyydä heitä tekemään päätös sen perusteella. Kun päätös ei onnistu, puuttuva tieto muuttuu näkyväksi. Purussa hyvä portti voidaan kuvata pienenä päätösnäkymänä: ihminen ymmärtää yhdellä silmäyksellä, mitä häneltä pyydetään, mihin ehdotus perustuu ja mitä eri valinnoista seuraa.

## Aktiviteetti 3 — Mitä tapahtuu hiljaisuudessa? (15 min)

Ryhmät määrittävät yhdelle rahaa koskevalle päätökselle:

- vastausajan
- varahenkilön
- eskalointiketjun
- turvallisen oletustoiminnon

Kriittistä päätöstä ei hyväksytä automaattisesti pelkän vastaamattomuuden vuoksi.

Pyydä ryhmiä perustelemaan erityisesti turvallinen oletustoiminto. Rahaa, oikeuksia tai ulkoista sitoumusta koskevassa tilanteessa odottaminen, hylkääminen tai eskalointi on yleensä perustellumpaa kuin automaattinen hyväksyntä. Samalla huomataan, että aikaraja ei ole pelkkä käyttöliittymän yksityiskohta vaan osa agentin ohjauskehyksen toimintalogiikkaa.

## Aktiviteetti 4 — Palautteesta hallituksi muutokseksi (20 min)

Täytä ketju:

| Vaihe | Kuvaus |
| --- | --- |
| Lokitettava päätös |  |
| Ihmisen perustelu |  |
| Toistuva ongelma, jota etsitään |  |
| Muutos, jota ehdotetaan |  |
| Kuka hyväksyy muutoksen? |  |
| Millä testillä vaikutus tarkistetaan? |  |

Korosta, ettei päätöksen tallentaminen vielä muuta kielimallia tai agentin toimintaa.

Käy yksi esimerkki koko ketjun läpi. Jos hyväksyjä hylkää samanlaisen ehdotuksen toistuvasti, loki paljastaa ilmiön, mutta sääntöä ei silti muuteta suoraan. Ensin ihminen arvioi, onko kyse aidosta toimintatavasta vai tilapäisestä poikkeuksesta. Vasta tämän jälkeen muutos tehdään ohjeeseen, agentin ohjauskehykseen tai testiin ja sen vaikutus tarkistetaan uudella testiajolla.

## Opettajan tarkistuslista

- Portit sijaitsevat riskikohdissa, eivät kaikkialla.
- Päätöksentekijä saa tarpeellisen taustatiedon.
- Aikarajan jälkeinen toiminta on määritelty.
- Palaute muuttuu toiminnaksi vain arvioidun kehitysprosessin kautta.
